「使われる」ための体験設計（UX / Adoption）

プロジェクト背景

あなたは 日本の中小企業向け・業務システムに精通したプロダクトデザイナー兼UXエンジニア です。
本プロジェクトは Ruby on Rails 8 で構築された マルチテナント型・共通業務基盤SaaS です。
	•	フェーズ1：基盤設計（認証・テナント・権限）
	•	フェーズ2：運用・拡張を楽にする（監査ログ・権限一致・管理画面）
	•	フェーズ3：実際に「使われる」体験を作る ← 今回

対象ユーザーは ITリテラシーが高くない中小企業の現場ユーザー です。

⸻

フェーズ3のゴール

以下をすべて満たすこと：
	1.	初回ログイン後、迷わず最初の行動が取れる
	2.	「次に何をすればいいか」が 常に画面から分かる
	3.	権限・役割に応じて 不要な情報は見せない
	4.	毎日／毎週 自然に使われ続ける導線 がある
	5.	管理者と一般ユーザーで 体験が破綻しない

⸻

作成してほしい成果物

① 体験設計の全体像（文章）
	•	「初回ログイン → 定常利用」までの体験フロー
	•	管理者 / 一般ユーザーそれぞれの体験差分
	•	「この瞬間に離脱しやすい」ポイントと対策

⸻

② 具体的なUX施策リスト（実装前提）

以下の観点で 実装可能なレベル まで落とし込むこと：
	•	オンボーディング（初回ガイド・初期データ）
	•	空状態（Empty State）の設計
	•	CTA（次にやるべき行動）の置き方
	•	通知・リマインド（やりすぎない）
	•	権限とUIの整合性（見える＝できる）
	•	エラー・未設定時のフォロー

⸻

③ Rails実装を意識した設計指針

以下を明確にすること：
	•	controller / view / helper / component の役割分担
	•	共通化すべきUIパターン
	•	can? / 権限を前提としたUX制御
	•	Tailwind CSS 前提のUI設計思想
	•	Turbo / Stimulus を使うべき箇所

⸻

④ 管理画面における「使われるUX」ルール
	•	管理画面でやってはいけないUX
	•	一覧・詳細・編集画面の理想構成
	•	保存・戻る・キャンセルの配置原則
	•	「設定したまま忘れられる」を防ぐ仕組み

⸻

⑤ フェーズ3の完了条件（Definition of Done）

以下をチェックリスト形式で定義すること：
	•	初見ユーザーが説明なしで操作できるか
	•	権限による違和感がないか
	•	毎日使う理由がUIに存在するか
	•	サポート問い合わせを減らせるか

⸻

制約条件（重要）
	•	日本の中小企業・現場利用が前提
	•	モバイル対応は 最低限（管理はPC想定）
	•	機能追加ではなく 体験改善が主目的
	•	「かっこいい」より「迷わない」を最優先
