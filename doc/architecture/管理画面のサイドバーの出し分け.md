あなたは Rails 8 の業務SaaS（マルチテナント/RBAC）実装に精通したエンジニアです。
既に can?(permission_key) ヘルパー（または Ability）が存在し、Permission の key は admin.<resource>.<action> 形式で管理されている前提です。
管理画面 /admin のサイドバーを 権限に応じて表示/非表示にしてください（見えるメニュー＝できること）。

---

ゴール
	•	app/views/admin/shared/_sidebar.html.erb のメニュー項目を can? で出し分け
	•	直URL対策は Controller 側（Admin::BaseController）で行われている前提。本タスクは UI 表示制御。
	•	サイドバーの項目は「ユーザーが実行できる action が1つも無い場合は表示しない」
	•	セクション（例：基本管理 / セキュリティ）も 配下が全部非表示ならセクションごと隠す
	•	TailwindCSS 4.1 のクラスのみで整える（JS不要）

---

前提（既存構成）
	•	管理レイアウト：app/views/layouts/admin.html.erb
	•	partial：
	•	app/views/admin/shared/_sidebar.html.erb（対象）
	•	app/views/admin/shared/_topbar.html.erb
	•	can?(key) が利用可能（AuthorizationHelper 等）
	•	ルーティングは namespace :admin 配下

---

表示制御ルール（必須）
	1.	メニュー表示条件
	•	例：テナント管理メニューは以下のいずれかを満たす場合のみ表示
	•	can?("admin.tenants.index")
	•	can?("admin.tenants.show")
	•	can?("admin.tenants.create")
	•	can?("admin.tenants.update")
	•	can?("admin.tenants.destroy")
	2.	リンク先は基本 index
	•	index 権限がないが show だけあるなど特殊ケースは、適切なリンク先（例：show へ）へ誘導するロジックを入れて良い
	•	ただし複雑にしすぎず、まずは index 権限の有無を優先
	3.	アクティブ表示
	•	現在のパス（request.path）や controller_name を用いて、現在ページのメニューを強調表示（Tailwindのクラス切替）
	4.	セクションのまとめ表示
	•	セクション見出し（例：基本管理/権限管理）は、配下リンクが1つも表示されない場合は見出しも出さない
	5.	拡張しやすさ
	•	メニュー定義は sidebar_items 的な配列/ハッシュで整理し、ERBにロジックを散らしすぎない
	•	可能なら app/helpers/admin/sidebar_helper.rb を作成して、表示可否判定や active 判定を集約

---

メニュー構成（例：最低限これを実装）

セクション：基本管理
	•	ダッシュボード（admin.dashboard.index）※権限キーは seed に合わせる
	•	テナント（admin.tenants.index）
	•	ユーザー（admin.users.index）

セクション：権限管理
	•	ロール（admin.roles.index）
	•	権限（admin.permissions.index）※一覧のみでもOK
	•	割り当て（admin.assignments.index）※なければ後回しOK（ただし定義枠は用意）

※ ルーティング/画面がまだ無いものは、リンクはコメントアウトせず「権限があっても route が無い」状態にならないよう、
存在しない場合はメニュー定義から外すか、respond_to? 的なガードを入れる（どちらでも良い）。プロジェクト方針として破綻しない案を選べ。

---

実装タスク（必須）
	1.	app/helpers/admin/sidebar_helper.rb（新規）
	•	sidebar_sections を返す（表示可能なものだけ）
	•	visible?(keys) のような「複数権限のOR判定」
	•	active?(path_or_controller) の判定
	2.	app/views/admin/shared/_sidebar.html.erb を更新
	•	helper を使ってループ描画
	•	Tailwindで見やすく（アイコンは不要、テキストだけでOK）
	3.	必要なら AuthorizationHelper の can? 呼び出しを helper 側から利用可能にする（include/依存整理）
	4.	影響があれば admin.html.erb 側も最小修正して sidebar を呼ぶ

---

出力形式（必須）
	•	変更/追加ファイルを ファイルパス付きで列挙
	•	それぞれ 全文コード
	•	想定する permission key 一覧（このサイドバーで参照するもの）も出す
	•	動作確認手順（例：仮で User に role を付与/外して sidebar が変わる）を簡潔に書く

---

受け入れ条件
	•	権限がないユーザーに切り替えると、サイドバーから該当項目が消える
	•	セクション配下が空ならセクション自体も消える
	•	現在表示中ページのメニューがアクティブ表示される
	•	ERBが複雑化せず、項目追加が1箇所の編集で済む（helperの定義追加程度）

この仕様に従って実装してください。
